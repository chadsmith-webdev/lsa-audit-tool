# Auth Implementation Plan — LSA Audit Tool
**Goal:** Add Supabase Auth (magic link) to gate the dashboard only. The audit tool remains fully public. Audits run while logged in get claimed by the authenticated user.

---

## Assumptions
- Supabase project already exists and is connected
- Site is live at `audit.localsearchally.com`
- `audits` table exists with `id` (UUID), `input` (jsonb), `result` (jsonb), `created_at`
- No `user_id` column on `audits` yet
- No middleware or session handling exists yet
- Auth will be passwordless magic link — no passwords

---

## Approach

Supabase Auth handles users, sessions, and JWTs natively. We add a `user_id` FK to `audits`, configure magic link redirects, protect `/dashboard` via Next.js middleware, and wire a minimal sign-in UI.

The audit route stays open. If a user is logged in when they run an audit, the resulting audit record gets their `user_id`. If not logged in, `user_id` is null — audit still saves and the shared link still works.

---

## Step-by-Step Plan

### Step 1 — Supabase project config (manual, you do this in the dashboard)
1. Go to Supabase > Authentication > URL Configuration
2. Set Site URL: `https://audit.localsearchally.com`
3. Add to Redirect URLs: `https://audit.localsearchally.com/auth/callback`
4. Go to Authentication > Email Templates > Magic Link — customize the email to match LSA brand voice if desired (optional)
5. Confirm email provider is enabled (it is by default)

### Step 2 — Database migration
Add `user_id` to the `audits` table and enable Row Level Security.

Run in Supabase SQL editor:

```sql
-- Add user_id column (nullable — existing audits stay intact)
ALTER TABLE audits ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE SET NULL;

-- Index for dashboard queries
CREATE INDEX idx_audits_user_id ON audits(user_id);

-- Enable RLS
ALTER TABLE audits ENABLE ROW LEVEL SECURITY;

-- Policy: user can read their own audits
CREATE POLICY "users_read_own_audits"
  ON audits FOR SELECT
  USING (auth.uid() = user_id);

-- Policy: service role can read/write all (for the API route using service key)
-- This is implicit when using the service_role key — no policy needed for that key

-- Policy: allow public insert (audit tool is open, no auth required to run one)
CREATE POLICY "public_insert_audits"
  ON audits FOR INSERT
  WITH CHECK (true);

-- Policy: allow public select by id (shared audit link still works unauthenticated)
CREATE POLICY "public_read_by_id"
  ON audits FOR SELECT
  USING (true);
```

Note: the last two policies mean RLS is permissive for reads/inserts — that's intentional for v1. The dashboard filters by `user_id` in the query, not solely via RLS. Tighten if needed later.

### Step 3 — Install Supabase Auth helpers
```bash
npm install @supabase/ssr
```
This is the current recommended package for Next.js App Router. Replaces the deprecated `@supabase/auth-helpers-nextjs`.

### Step 4 — Supabase client updates (`lib/supabase.ts`)
Create two client factories:
- `createBrowserClient()` — for Client Components (uses anon key, reads session from cookies)
- `createServerClient(cookieStore)` — for Server Components and middleware (reads session server-side)

The existing `getSupabase()` uses the service role key for server-side DB writes — keep that as-is for API routes that need to bypass RLS.

### Step 5 — Auth callback route (`app/auth/callback/route.ts`)
Magic links redirect to `/auth/callback?code=...`. This route exchanges the code for a session and redirects the user.

```
app/
  auth/
    callback/
      route.ts   ← exchanges code, sets session cookie, redirects to /dashboard
```

Logic:
1. Read `code` from search params
2. Call `supabase.auth.exchangeCodeForSession(code)`
3. Redirect to `/dashboard` on success, `/login?error=1` on failure

### Step 6 — Middleware (`middleware.ts` at project root)
Protects all `/dashboard` routes. Public routes pass through untouched.

```
Protected: /dashboard, /dashboard/*
Public: everything else (/, /audit/*, /api/*, /auth/*)
```

Logic:
1. Create server Supabase client from request cookies
2. Call `supabase.auth.getUser()` — refreshes session if expired
3. If no user and path starts with `/dashboard`, redirect to `/login`
4. Always forward the response with updated session cookies

### Step 7 — Login page (`app/login/page.tsx`)
Minimal. Email input, submit button, confirmation state.

- Server Component wrapper, Client Component for the form
- On submit: call `supabase.auth.signInWithOtp({ email, options: { emailRedirectTo: 'https://audit.localsearchally.com/auth/callback' } })`
- On success: show "Check your email — we sent you a link" message
- No password field, no "create account" — magic link only
- Style matches existing dark design system (bg #0a0a0a, carolina blue CTA)
- If user is already logged in (has session), redirect to `/dashboard`

### Step 8 — Attach user_id on audit creation (`app/api/audit/route.ts`)
When an audit is saved to Supabase, check if a session exists on the request and attach `user_id` if so.

- Parse the `Authorization` header or session cookie from the request
- If valid session found: include `user_id` in the insert payload
- If no session: insert with `user_id: null` (audit still saves, shared link works)

This uses the server Supabase client (anon key + cookies), not the service role client, so Supabase Auth can verify the session.

### Step 9 — Dashboard scaffold (`app/dashboard/page.tsx`)
Minimal first version — just proves auth is working end-to-end.

- Server Component
- Gets session via server Supabase client
- Queries `audits` where `user_id = auth.uid()`, ordered by `created_at DESC`
- Renders a list of the user's audits with links to the shared audit view
- Sign out button

This is the foundation the geo-grid and monitoring features get added to.

## Risks & Notes
- RLS policies are permissive for v1 — public read/insert is intentional. Revisit when dashboard becomes a paid product.
- `@supabase/ssr` requires careful cookie handling in middleware — must forward the response object with updated cookies or sessions won't persist. This is the most common mistake with Supabase + App Router.
- Magic link emails may land in spam for some providers on first send. Supabase recommends setting up a custom SMTP sender (Resend) for production — you already have Resend in this project, so that's a straightforward config.
- The audit API route currently uses the service role key for DB writes. Attaching `user_id` requires a second client call (anon key + cookies) just to read the session — both clients will be used in that route after Step 8.
